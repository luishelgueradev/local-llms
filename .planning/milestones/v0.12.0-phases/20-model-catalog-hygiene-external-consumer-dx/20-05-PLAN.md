4. **`disabled: true` flag reference (CAT-01):**

```yaml
# router/models.yaml — example
- name: qwen2.5-7b-instruct-q4km
  backend: llamacpp
  # ... existing fields ...
  disabled: true   # Phase 20 / CAT-01 / D-01 — backend not running on this host
```

Semantics:
- Disabled entries do NOT appear in GET /v1/models or GET /v1/models/:id (404 model_not_found)
- registry.resolve(disabled_name) throws RegistryUnknownModelError (same envelope as a fully-unknown alias — anti-leak)
- VRAM envelope superRefine skips disabled entries (re-enable by flipping flag does not retro-trigger validation)
- Re-enable: change `disabled: true` → `disabled: false` (or delete the line) + Valkey DEL + force-recreate per the hot-edit recipe

5. **`health` field reference (CAT-02):**

Shape:
```json
"health": {
  "status": "ok" | "degraded" | "down" | "unknown",
  "checked_at": "2026-06-03T04:00:00.000Z"
}
```

- **Boot-time probe**: each declared backend (from enabled entries) probed once on app.ready
- **Lazy refresh**: triggered on the next /v1/models request after `ROUTER_BACKEND_HEALTH_TTL_SEC` (default 60) seconds since the cached `checked_at`
- **NO auto-filter** (D-04 LOCKED): a `down` entry still appears in the response — the consumer decides
- **ollama-cloud** always reports `unknown` — no public /healthz the bearer can hit. This is honest, not a bug.

6. **`recommendations:` block (CDX-01):**

```yaml
# router/models.yaml — top level
recommendations:
  chat-local-default: chat-local
  chat-json-strict-default: chat-local      # consumer reads this for json_mode picks
  embed-default: embed-local
  rerank-default: bge-reranker-local
  vision-default: vision-local
```

- Operator-configurable; values cross-field-validated against enabled (non-disabled) model names at boot
- When omitted, router auto-derives from per-entry `recommended_for` tags (first matching enabled entry wins per profile)
- Naming convention: `<use-case>-<profile>-default` where profile ∈ {local, cloud}

7. **`deprecated_aliases:` block (CAT-04):**

```yaml
# router/models.yaml — top level
deprecated_aliases:
  qwen2.5-7b-instruct-q4km:
    target: chat-local
    deprecated_since: v0.12.0
    removal_target: v0.13.0
```

- Dispatch-time resolution: a request for the deprecated alias is REDIRECTED to the canonical target's entry
- Response carries `X-Deprecated-Alias: <canonical>` header
- Structured pino warn log per call: `{event: 'deprecated_alias_used', alias, redirected_to, ...}`
- Prometheus counter `router_deprecated_alias_used_total{old_name, new_name}` (POL-06 — `_id` suffix forbidden)
- Removal is operator-side: when ready to actually break, delete the alias's row from `deprecated_aliases:` and the disabled entry from `models:`
