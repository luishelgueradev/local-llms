# Consumer Migration Guide — v0.12.0

**Audience:** Anyone running an external project that consumes `local-llms` via its OpenAI/Anthropic API surface (artiscrapper, n8n workflows at `objetiva.com.ar`, Unsloth Studio, Open WebUI, custom agents/scripts).

**TL;DR:**
- **Nothing breaks.** Existing aliases continue to resolve. Existing response shapes are unchanged for all fields you currently read.
- **Three quant-encoded aliases are deprecated** (still resolve until v0.13.0). If you call them, your responses now carry `X-Deprecated-Alias: <canonical>` — update at your leisure within the 30+ day grace window.
- **Three new optional features** are available for programmatic alias selection: `health` field on `/v1/models`, `recommendations` map on `/v1/models`, and a new public `/version` endpoint.

---

## 1. What Did NOT Change (you can stop reading here if you want)

| Surface | v0.11.0 | v0.12.0 |
|---------|---------|---------|
| `POST /v1/chat/completions` request/response shape | OpenAI-compatible | OpenAI-compatible (additive `X-Deprecated-Alias` header on deprecated-alias calls only) |
| `POST /v1/messages` (Anthropic) | unchanged | unchanged |
| `POST /v1/embeddings` request/response shape | Byte-identical SHA (P7-01 invariant) | **Byte-identical SHA preserved** (Phase 20 does not touch the embeddings translator) |
| `POST /v1/responses` streaming | RESS-01..05 contract | unchanged |
| `POST /v1/rerank` | unchanged | unchanged |
| `GET /v1/models` existing fields (`id`, `object`, `owned_by`, `capabilities`, `policy.cloud_allowed`) | unchanged | **unchanged** (additive `health` + `recommended_for` + top-level `recommendations` only) |
| `GET /v1/models/:id` existing fields | unchanged | **unchanged** (additive `health` + `recommended_for` only) |
| `/mcp` Streamable HTTP surface (Phase 15) | unchanged | unchanged |
| Bearer auth model | single token in `.env` | unchanged |
| Canonical semantic aliases (`chat-local`, `embed-local`, `big-cloud`, `vision-local`, `bge-reranker-local`) | resolve as expected | **resolve identically** |

If you use only the canonical semantic aliases above, your migration is complete — no code changes required.

---

## 2. Deprecated Aliases (action recommended before v0.13.0)

Three legacy aliases that encode the BACKEND in the model name are now **deprecated**. They still resolve, but redirect to a canonical semantic equivalent. Each redirect:
- Returns a successful response from the canonical target
- Sets the HTTP response header `X-Deprecated-Alias: <canonical>` so you can detect programmatically
- Emits a structured warn log on the router side (`event: 'deprecated_alias_used'`)
- Increments the Prometheus counter `router_deprecated_alias_used_total{old_name, new_name}` (operator-side visibility)

Migration table:

| Deprecated alias | Canonical replacement | Deprecated since | Removal target |
|------------------|----------------------|------------------|----------------|
| `qwen2.5-7b-instruct-q4km` | `chat-local` | v0.12.0 | v0.13.0 |
| `qwen2.5-7b-instruct-awq` | `chat-local` | v0.12.0 | v0.13.0 |
| `bge-m3-vllm` | `embed-local` | v0.12.0 | v0.13.0 |

**Detect deprecation programmatically:**

```javascript
// JavaScript / TypeScript
const res = await fetch('http://localhost:3210/v1/chat/completions', { ... });
const deprecated = res.headers.get('X-Deprecated-Alias');
if (deprecated) {
  console.warn(`Used deprecated alias; canonical: ${deprecated}. Update your client.`);
}
```

```python
# Python (httpx)
r = httpx.post('http://localhost:3210/v1/chat/completions', json=...)
if 'x-deprecated-alias' in r.headers:
    logger.warning(f"Deprecated alias used; canonical: {r.headers['x-deprecated-alias']}")
```

**Per-consumer migration steps:**

- **n8n workflows** — open each workflow that references one of the deprecated alias names in an "OpenAI Chat Model" or "OpenAI" credential node; change the model field to the canonical replacement. Save + activate. No router-side action needed.
- **Unsloth Studio** — open the model picker, switch from the deprecated name to the canonical. The router's `/v1/models` will already show only the canonical (deprecated entries are invisible at the surface per the Phase 20 disabled-flag).
- **artiscrapper / custom agents** — find references to `qwen2.5-7b-instruct-q4km`, `qwen2.5-7b-instruct-awq`, `bge-m3-vllm` in your codebase and replace with the canonical name. Re-deploy.
- **Open WebUI** — server-to-server connection lists are derived from `/v1/models` on each call; deprecated aliases will not appear, so no UI-side action is needed.

**Hard removal:** v0.13.0 will remove the deprecation map; calls to the old names will return `404 model_not_found` instead of redirecting. The exact v0.13.0 ship date is operator-discretion — read the milestone announcement in `.planning/PROJECT.md` when v0.13.0 is opened.

## 3. New Optional Features (adopt at your own pace)

### 3.1 `health` field on `/v1/models` entries

Each entry now includes a `health` object:

```json
{
  "id": "chat-local",
  "object": "model",
  "owned_by": "local-llms",
  "capabilities": ["chat","tools","json_mode"],
  "policy": {"cloud_allowed": true},
  "health": {
    "status": "ok",
    "checked_at": "2026-06-03T04:00:00.000Z"
  },
  "recommended_for": ["chat","chat-tools","chat-json-strict","function-calling"]
}
```

`status` ∈ `'ok' | 'degraded' | 'down' | 'unknown'`. `ollama-cloud`-backed entries always report `'unknown'` (no public `/healthz` the router's bearer can probe — this is honest, not a bug).

Use it to skip calls to backends that are currently down:

```bash
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:3210/v1/models \
  | jq '.data[] | select(.health.status == "ok") | .id'
```

### 3.2 `recommendations` map on `/v1/models`

A new top-level field returns the operator-declared recommendation map:

```json
{
  "object": "list",
  "data": [ ... ],
  "recommendations": {
    "chat-local-default": "chat-local",
    "chat-json-strict-default": "chat-local",
    "embed-default": "embed-local",
    "rerank-default": "bge-reranker-local",
    "vision-default": "vision-local"
  }
}
```

Programmatic alias selection — the artiscrapper use case ("chat + json_mode + local + working right now") becomes:

```bash
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:3210/v1/models | jq -r '
  .recommendations["chat-json-strict-default"] as $alias
  | .data[] | select(.id == $alias and .health.status == "ok") | .id
'
```

Returns `chat-local` when healthy; empty when not — your client falls back accordingly.

### 3.3 `X-Deprecated-Alias` response header

Documented in §2 above. Adopt this if you want to surface deprecation warnings in your client logs proactively.

### 3.4 New `/version` endpoint (public, no bearer)

```bash
curl -s http://localhost:3210/version | jq
# {
#   "build_sha": "abc1234...",
#   "build_time": "2026-06-03T04:00:00Z",
#   "node_version": "v22.x.x",
#   "git_dirty": false
# }
```

Use case: detect router image drift in CI ("did my hot-fix actually deploy?"). Operators use `bash bin/deploy-router.sh check` to wrap this — see [DEPLOY.md → Model Catalog Hygiene](../DEPLOY.md#model-catalog-hygiene-phase-20--v0120).

### 3.5 `/healthz` additive fields

`build_sha`, `build_time`, `node_version`, `git_dirty` were added (additive — existing `status` / `service` / `phase` / `registry_models` unchanged).

## 4. Operator Surface (FYI for consumers)

The operator now has:
- A `disabled: true` flag on `router/models.yaml` entries (catalog hygiene)
- A `recommendations:` block (consumer recommendations)
- A `deprecated_aliases:` block (operator-declared deprecation map)
- A `bin/deploy-router.sh` script with `full / config-only / check` subcommands

You as a consumer do not interact with these — but if your stack adds new aliases or removes deprecated ones, your operator handles it without breaking your client (as long as you use the recommendations map for alias discovery).
