<objective>
Create the **GPU passthrough preflight** that catches WSL2's silent CPU-fallback failure mode (PITFALLS Pitfall 1) before any GPU service starts. This script is the single most load-bearing piece of Phase 1: it asserts that NVIDIA Container Toolkit is functional and records driver state that Phase 7 will read to pick the correct vLLM image tag.

Purpose: WSL2 + NVIDIA Container Toolkit fails silently — containers start, models "load," but inference runs on CPU at 1/100th the expected throughput. A preflight that fails closed (and is invoked by Compose via `service_completed_successfully` in Plan 03) makes this an upfront, actionable error instead of a multi-day debug rabbit hole.

Output: `bin/preflight-gpu.sh` (executable bash). On every run it writes `/srv/local-llms/.preflight-state.json` with driver/CUDA/nvidia-ctk versions + per-check pass/fail. The same script is mounted into the `gpu-preflight` Compose service in Plan 03 and serves as the in-container preflight too.

Implements user decisions: D-05 (5 checks + record driver state), D-06 (pinned `nvidia/cuda:12.6.0-base-ubuntu24.04` for the container check), D-07 (state file at `/srv/local-llms/.preflight-state.json`, schema is a contract Phase 7 reads).
</objective>

Create `bin/preflight-gpu.sh` as a `bash` script. Make it executable: `chmod +x bin/preflight-gpu.sh` after writing.

Required behavior (D-05, D-06, D-07):

1. **Shebang and strict mode:**
   ```bash
   #!/usr/bin/env bash
   set -uo pipefail
   # NOTE: do NOT use `set -e` — we want to run all 5 checks even if early ones fail,
   # so the state file records the full picture. Track failures via a counter.
   ```

2. **CLI flags (optional but cheap):**
   - `-q` / `--quiet`: only print the summary line.
   - `-v` / `--verbose`: print every check command + its output.
   - Default: print pass/fail per check + summary.

3. **Resolve `HOST_DATA_ROOT` exactly like `bin/bootstrap-host.sh` did:**
   - Read from `.env` if present in repo root; fall back to `/srv/local-llms`.
   - Resolve via `SCRIPT_DIR="$( cd "$( dirname "${BASH_SOURCE[0]}" )" && pwd )"; REPO_ROOT="$(dirname "$SCRIPT_DIR")"`.

4. **Detect run mode (host vs in-container):**
   - When the script is run on the host: it has access to `nvidia-smi`, `nvidia-ctk`, `/etc/docker/daemon.json`, and `docker`.
   - When the script is run **inside** the `gpu-preflight` Compose service (Plan 03 will mount it in): it has access to `nvidia-smi` (via NVIDIA Container Toolkit), but `nvidia-ctk`, `/etc/docker/daemon.json`, and the Docker CLI may not be present.
   - Detection heuristic: presence of `/.dockerenv` indicates running in a container.
   - When in-container: skip the "host nvidia-smi", "nvidia-ctk --version", and "daemon.json" checks (those are validated from the host invocation), and only verify `/dev/dxg` (or `/dev/nvidia*`) exists (the Linux GPU device — `/dev/dxg` for WSL2, `/dev/nvidia0` etc. for native Linux), plus container `nvidia-smi`. Exit 0 if both pass.
   - When on host: run all 5 checks.

5. **Implement the 5 checks (D-05) on host runs. Each check returns 0/1 and updates two arrays — `CHECK_NAMES` and `CHECK_RESULTS`:**

   **Check 1 — `/dev/dxg` exists (WSL2) OR `/dev/nvidia*` exists (native Linux):**
   ```bash
   check_gpu_device() {
     if [ -e /dev/dxg ] || ls /dev/nvidia* >/dev/null 2>&1; then
       return 0
     fi
     return 1
   }
   ```

   **Check 2 — host `nvidia-smi` works and reports a GPU:**
   ```bash
   check_host_nvidia_smi() {
     command -v nvidia-smi >/dev/null 2>&1 || return 1
     nvidia-smi --query-gpu=name --format=csv,noheader 2>/dev/null | head -1 | grep -q '.' || return 1
     return 0
   }
   ```

   **Check 3 — container `nvidia-smi` works (proves NVIDIA Container Toolkit is functional). MUST use the pinned `nvidia/cuda:12.6.0-base-ubuntu24.04` image (D-06):**
   ```bash
   check_container_nvidia_smi() {
     command -v docker >/dev/null 2>&1 || return 1
     docker run --rm --gpus all nvidia/cuda:12.6.0-base-ubuntu24.04 nvidia-smi 2>/dev/null \
       | grep -q 'NVIDIA-SMI' || return 1
     return 0
   }
   ```

   **Check 4 — `nvidia-ctk --version` works:**
   ```bash
   check_nvidia_ctk() {
     command -v nvidia-ctk >/dev/null 2>&1 || return 1
     nvidia-ctk --version 2>/dev/null | grep -qiE 'version|nvidia-ctk' || return 1
     return 0
   }
   ```

   **Check 5 — `/etc/docker/daemon.json` has the `nvidia` runtime entry:**
   ```bash
   check_daemon_json() {
     local f=/etc/docker/daemon.json
     [ -f "$f" ] || return 1
     # The runtime entry is registered by `nvidia-ctk runtime configure --runtime=docker`.
     # Look for a "nvidia" key inside a "runtimes" object.
     python3 -c "
     import json, sys
     try:
       d = json.load(open('$f'))
       sys.exit(0 if 'nvidia' in (d.get('runtimes') or {}) else 1)
     except Exception:
       sys.exit(1)
     " 2>/dev/null && return 0
     # Fallback to grep if python3 is not available
     grep -qE '\"nvidia\"\\s*:' "$f" && return 0
     return 1
   }
   ```

6. **Record driver state (D-05 + D-07):**
   Capture the following BEFORE writing the state file. Each capture must be defensive — if a tool isn't present, record the field as `null`, never abort.

   - `host_driver_version`: from `nvidia-smi --query-gpu=driver_version --format=csv,noheader 2>/dev/null | head -1` (e.g. `"555.85"`)
   - `cuda_version`: from `nvidia-smi --query | grep -oE 'CUDA Version: [0-9]+\\.[0-9]+' | head -1 | awk '{print $3}'` (e.g. `"12.6"`)
   - `nvidia_ctk_version`: from `nvidia-ctk --version 2>/dev/null | head -1` (full first line)
   - `last_run_at`: from `date -u +%Y-%m-%dT%H:%M:%SZ` (ISO 8601 UTC)
   - `host_kernel`: from `uname -r` (handy for WSL2 vs native debugging; small, free, useful)
   - `wsl2`: boolean — `true` if `/proc/version` contains `microsoft` or `WSL`, else `false`

7. **Write the state file at `${HOST_DATA_ROOT}/.preflight-state.json`. Schema (D-07 — set in stone, Phase 7 contract):**
   ```json
   {
     "schema_version": 1,
     "last_run_at": "2026-05-10T15:42:08Z",
     "host_driver_version": "555.85",
     "cuda_version": "12.6",
     "nvidia_ctk_version": "NVIDIA Container Toolkit CLI version 1.16.2",
     "host_kernel": "5.15.146.1-microsoft-standard-WSL2",
     "wsl2": true,
     "checks": {
       "gpu_device": "pass",
       "host_nvidia_smi": "pass",
       "container_nvidia_smi": "pass",
       "nvidia_ctk": "pass",
       "daemon_json": "pass"
     },
     "passed": true
   }
   ```
   - When running in-container, only the keys `gpu_device` and `container_nvidia_smi` are populated under `checks`; other check keys are set to `"skipped"` and `host_*` and `nvidia_ctk_version` are set to the special string `"in-container"` (the host-side fields aren't accessible from within the container — preserve the schema, signal the run mode).
   - `passed` is `true` only if ALL non-skipped checks pass.
   - Use `python3 -c 'import json; print(json.dumps(...))'` to write the file (jq is not always present; python3 is). If neither is available, hand-build the JSON via printf — no `awk`/`sed` JSON construction (fragile).
   - Write to a temp path then `mv` for atomicity:
     ```bash
     TMPFILE="${STATE_FILE}.tmp.$$"
     ... write to "$TMPFILE" ...
     mv "$TMPFILE" "$STATE_FILE"
     ```

8. **Print summary (Claude's discretion on log format):**
   ```
   [preflight] Check                  Result
   [preflight] gpu_device              PASS
   [preflight] host_nvidia_smi         PASS
   [preflight] container_nvidia_smi    PASS
   [preflight] nvidia_ctk              PASS
   [preflight] daemon_json             PASS
   [preflight] -----------------------------
   [preflight] All checks passed. State written to /srv/local-llms/.preflight-state.json
   ```
   On failure, print which checks failed AND a remediation hint per failed check:
   - `gpu_device`: "On WSL2, ensure the Windows-side NVIDIA driver is installed. NEVER install a Linux NVIDIA driver inside the WSL distro."
   - `host_nvidia_smi`: "Install / verify the host NVIDIA driver. On WSL2, this is on Windows."
   - `container_nvidia_smi`: "Install NVIDIA Container Toolkit and run: sudo nvidia-ctk runtime configure --runtime=docker && sudo systemctl restart docker"
   - `nvidia_ctk`: "Install nvidia-container-toolkit (apt install nvidia-container-toolkit)."
   - `daemon_json`: "Run: sudo nvidia-ctk runtime configure --runtime=docker && sudo systemctl restart docker"

9. **Exit code (D-04 contract — Compose's `service_completed_successfully` reads exit 0):**
   - `0` if all non-skipped checks pass.
   - `1` if any check fails.
   - The state file is written either way (D-07: "on every run, pass or fail").

**Forbidden:**
- Do NOT install anything (`apt install`, `curl | sh`, `nvidia-ctk runtime configure`) — preflight is read-only verification, NOT a remediation.
- Do NOT modify `daemon.json`, `.env`, or any config — preflight only reads.
- Do NOT use `:latest` for the container check image — it MUST be `nvidia/cuda:12.6.0-base-ubuntu24.04` (D-06).
- Do NOT skip the WSL2 device check just because `/dev/nvidia*` exists too — `[ -e /dev/dxg ] || ls /dev/nvidia* >/dev/null 2>&1` covers both.
- Do NOT add a `--fix` flag or any auto-remediation — that violates the read-only contract.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Preflight script reads `/etc/docker/daemon.json` (host) | Read-only; the file is world-readable on most installs anyway. |
| Preflight runs `docker run --rm --gpus all nvidia/cuda:12.6.0-base-ubuntu24.04 nvidia-smi` | Spawns a short-lived container with GPU access. Image is pinned (D-06). The user must already have docker socket access; preflight does not escalate. |
| Preflight writes `/srv/local-llms/.preflight-state.json` | The state file lives under HOST_DATA_ROOT, owned by the invoking user (Plan 01 Task 1 sets ownership). No secrets are written — only driver/CUDA/nvidia-ctk versions and per-check pass/fail. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | The pinned `nvidia/cuda:12.6.0-base-ubuntu24.04` image could be replaced upstream | accept | NVIDIA's CUDA images on Docker Hub are signed-by-publisher; pinned by full tag. Image content drift on a specific tag is rare in the NVIDIA registry. Phase 9 ops hardening can revisit periodic pin updates. |
| T-02-02 | Information disclosure | State file contains driver version | accept | Driver version is not a secret. The file is owned by the invoking user (mode 644 acceptable, 600 if HOST_DATA_ROOT permissions enforce). No bearer tokens or PII in the state file. |
| T-02-03 | Denial of service | Preflight container hangs (network failure during image pull on first run) | mitigate | The script is invoked manually (host) or as a Compose one-shot (Plan 03). On first compose up, image pull will block — that's expected behavior. A `--timeout 60` flag could be added but is not required for Phase 1; the image is small (~150 MB). |
| T-02-04 | Elevation of privilege | `docker run --gpus all` requires docker socket access | accept | Docker socket access is already implied by the project (every step requires `docker compose up`). Preflight does not escalate beyond what the user already has. The script does not write to `/etc/docker/daemon.json` (only reads). |
| T-02-05 | Spoofing | An attacker replaces `bin/preflight-gpu.sh` to silently pass | mitigate | Script is in-tree, code-reviewed via PRs. Compose `gpu-preflight` service in Plan 03 mounts the script read-only from the repo, so the running script == the committed script. Out-of-band tampering of the host script is bounded by host filesystem perms. |
</threat_model>
