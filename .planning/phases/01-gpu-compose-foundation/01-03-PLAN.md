**The full file (lift verbatim, this is the contract):**

```yaml
# compose.yml — local-llms platform
# Phase 1: GPU + Compose Foundation
#
# Adds: gpu-preflight (one-shot), ollama (single GPU consumer).
# Future phases append services to this same file (D-12 — no split files).
# Compose `profiles:` per-backend lands in Phase 3 (D-11).
#
# All services use:
#   - pinned image tags (no :latest — INFRA-04)
#   - the x-gpu YAML anchor for GPU reservation (INFRA-02)
#   - the four declared networks (D-13)
#   - depends_on: { gpu-preflight: { condition: service_completed_successfully } }
#     for any GPU consumer (D-04, INFRA-05)

name: ${COMPOSE_PROJECT_NAME:-local-llms}

# ── GPU reservation anchor — referenced by every GPU service ──────────────────
# DO NOT use `runtime: nvidia` (legacy form, on the standing rejection list).
# DO NOT use `gpus: all` shorthand (Compose-version-fragile).
# This block is the canonical form per Docker Compose v2 docs and PITFALLS Pitfall 2.
x-gpu: &gpu
  deploy:
    resources:
      reservations:
        devices:
          - driver: nvidia
            count: all
            capabilities: [gpu, utility]
  # `utility` capability is required for nvidia-smi to work INSIDE the container
  # (PITFALLS Pitfall 2). Without it the GPU works but you cannot introspect.

# ── Four networks (D-13) ──────────────────────────────────────────────────────
# Names invented in Phase 1; later phases attach to existing networks, never
# invent new ones. Phase 1 actually wires only `backend` (Ollama is the only
# member); `edge`, `app`, `data` are declared empty so future phases inherit
# the topology without touching this section.
networks:
  edge:
    # Public-facing — Traefik publishes ports here (Phase 6).
  app:
    # Router ↔ webui (Phase 2 attaches router; Phase 6 attaches webui).
  backend:
    internal: true
    # Router ↔ runtimes. internal: true blocks outbound internet from members,
    # which is correct for Ollama, llama.cpp, vLLM (no telemetry, no exfil).
  data:
    internal: true
    # Router/webui ↔ postgres/valkey (Phase 5 attaches Postgres; Phase 8 Valkey).

services:

  # ── GPU preflight (D-04, D-06, INFRA-01) ──────────────────────────────────
  # One-shot service that runs bin/preflight-gpu.sh from inside a pinned NVIDIA
  # CUDA base image. Exits 0 only if all preflight checks pass. Every GPU
  # service `depends_on` it with `condition: service_completed_successfully`,
  # so a failed preflight blocks the entire stack.
  gpu-preflight:
    image: nvidia/cuda:12.6.0-base-ubuntu24.04
    container_name: ${COMPOSE_PROJECT_NAME:-local-llms}-gpu-preflight
    <<: *gpu
    # Install bash + python3 (the Ubuntu 24.04 base has them, but be defensive).
    # The `bash` is needed because preflight-gpu.sh uses bash-only features.
    # The `python3` is used by preflight-gpu.sh to write the state file as JSON.
    # Both are present in nvidia/cuda:12.6.0-base-ubuntu24.04 — the entrypoint
    # below assumes that. If a future CUDA base drops them, the entrypoint can
    # `apt-get update && apt-get install -y --no-install-recommends bash python3`
    # at the cost of network and time. For Phase 1 we rely on the base image.
    entrypoint: ["/usr/bin/bash"]
    command: ["/preflight/preflight-gpu.sh"]
    volumes:
      # Mount the host-side preflight script read-only.
      - ./bin/preflight-gpu.sh:/preflight/preflight-gpu.sh:ro
      # Mount the host data root so the state file lands at the canonical path
      # (HOST_DATA_ROOT/.preflight-state.json — Phase 7 reads from here).
      - ${HOST_DATA_ROOT:-/srv/local-llms}:${HOST_DATA_ROOT:-/srv/local-llms}
    environment:
      # Tell preflight-gpu.sh where the host data root is from inside the container.
      # The script's `.env`-resolution logic also has a default fallback.
      - HOST_DATA_ROOT=${HOST_DATA_ROOT:-/srv/local-llms}
    # No restart policy — this is a one-shot. Compose's default is `no` which
    # is what we want.
    restart: "no"
    networks:
      - backend
    # No host port published — preflight only needs GPU + filesystem.

  # ── Ollama (D-11, BCKND-01) ───────────────────────────────────────────────
  ollama:
    image: ollama/ollama:0.5.7
    container_name: ${COMPOSE_PROJECT_NAME:-local-llms}-ollama
    <<: *gpu
    restart: unless-stopped
    # Phase 1 ships Ollama on the backend network only. The router (Phase 2)
    # will attach to backend AND app to talk to Ollama. Phase 1 publishes a
    # localhost-only host port so the smoke test (Plan 04) can hit the API
    # without going through a router that doesn't exist yet. After Phase 2
    # lands the router, this `ports:` block can be REMOVED — Ollama should
    # then only be reachable via the backend network from the router. The
    # localhost-only bind keeps the surface off the LAN even before the router.
    ports:
      - "127.0.0.1:11434:11434"
    networks:
      - backend
    volumes:
      # Bind-mount Ollama's blob store to the host two-volume layout.
      # /root/.ollama is Ollama's $OLLAMA_MODELS / writable home inside the
      # container. The host side lives under models-gguf/ollama (D-02 layout).
      # This is the half of the two-store split that backs Ollama; future
      # llama.cpp goes to models-gguf/gguf, vLLM to models-hf/.
      - ${HOST_DATA_ROOT:-/srv/local-llms}/models-gguf/ollama:/root/.ollama
    environment:
      # Bind to all interfaces inside the container so the backend network can
      # reach it. Default in the official image, but explicit > implicit.
      - OLLAMA_HOST=0.0.0.0:11434
      # Cap concurrent loaded models at 1 (16 GB VRAM constraint per
      # CLAUDE.md). PITFALLS Pitfall 3 — VRAM thrashing prevention.
      - OLLAMA_MAX_LOADED_MODELS=1
      # Cap parallel requests per loaded model. KV cache scales with this;
      # PITFALLS Pitfall 3 + Performance Traps row "OLLAMA_NUM_PARALLEL".
      - OLLAMA_NUM_PARALLEL=2
    healthcheck:
      # /api/tags is Ollama's "I'm alive" endpoint — returns the loaded model
      # list. Lightweight, doesn't touch GPU. Phase 1's Ollama image is
      # ollama/ollama:0.5.7 which ships with curl, so this works.
      test: ["CMD-SHELL", "curl -fsS http://127.0.0.1:11434/api/tags || exit 1"]
      interval: 10s
      timeout: 3s
      start_period: 30s
      retries: 5
    depends_on:
      # D-04: Compose-native preflight gating. If gpu-preflight exits non-zero,
      # ollama does not start.
      gpu-preflight:
        condition: service_completed_successfully
```

**Constraints / forbidden items:**

1. **No `:latest` tags anywhere.** `grep ':latest' compose.yml` must return empty.
2. **No `runtime: nvidia` (legacy form).** The `<<: *gpu` merge key + `deploy.resources.reservations.devices` block IS the only GPU access path.
3. **No `gpus: all` shorthand.**
4. **No `version:` top-level key.** Compose v2 dropped it; including it produces a deprecation warning. (The `name:` top-level key replaces `version:` and `COMPOSE_PROJECT_NAME` env handling.)
5. **No services for Phase 2+** — Phase 1 ships ONLY `gpu-preflight` and `ollama` per D-11. Do NOT pre-stub `router`, `llamacpp`, `vllm`, `postgres`, `valkey`, `traefik`, or `open-webui`. They land in their own phases.
6. **No `profiles:` keys.** Compose `profiles:` lands in Phase 3 (per D-11 and the deferred-ideas list).
7. **No published ports on `gpu-preflight`** — it's an internal one-shot.
8. **No published ports on `ollama` other than `127.0.0.1:11434:11434`** — the localhost-only bind is deliberate (off-LAN, available for the local smoke test in Plan 04).
9. **Network membership for Ollama is `backend` ONLY** — not `edge`, not `app`, not `data`. The router (Phase 2) is the boundary that crosses networks.
10. **Bind mount path uses `${HOST_DATA_ROOT:-/srv/local-llms}/models-gguf/ollama`**, never a Docker named volume. Per D-01 (in-tree NOT, named volumes NOT, host path YES).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host filesystem (`${HOST_DATA_ROOT}`) ↔ Ollama container (`/root/.ollama`) | Bind mount; container can write blob files into the host path. Ownership matched to invoking user by Plan 01's bootstrap. |
| Host network ↔ Ollama (port 11434) | Localhost-only bind (`127.0.0.1:11434:11434`) — not reachable from LAN. Phase 1 has no auth on Ollama; the router (Phase 2) adds bearer auth. |
| `gpu-preflight` container ↔ host script (`bin/preflight-gpu.sh`) | Script mounted read-only — container cannot tamper with it. |
| Ollama container ↔ outside world | `backend: internal: true` blocks all egress. Ollama is air-gapped from the internet. The router (Phase 2) will be on `backend` + `app` and will be the only egress path. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | An attacker on the LAN connects to Ollama port 11434 directly | mitigate | Port published as `127.0.0.1:11434:11434` (localhost-only). LAN cannot reach it. After Phase 2 lands the router, the port mapping should be removed entirely (only the router talks to Ollama via backend network). |
| T-03-02 | Tampering | The pinned Ollama image is replaced upstream | accept | Pinned tag `ollama/ollama:0.5.7`. Image content drift on a specific tag is rare. Phase 9 ops hardening can revisit periodic pin updates. |
| T-03-03 | Information disclosure | The model blob store is readable by anyone with shell access on the host | accept | The host data root is owned by the invoking user (Plan 01 chown). Models are public model weights — not secret. No PII in the blob store at this stage. |
| T-03-04 | Denial of service | First `docker compose up` triggers a 150 MB pull of `nvidia/cuda:12.6.0-base-ubuntu24.04` and a ~700 MB pull of `ollama/ollama:0.5.7` | accept | Expected behavior. Documented in README. After first pull, images are cached. Future Phase 9 could pre-pull in a wrapper script. |
| T-03-05 | Elevation of privilege | The `gpu-preflight` service has access to GPU and the host filesystem under HOST_DATA_ROOT | mitigate | Container is one-shot (`restart: "no"`), runs only `bin/preflight-gpu.sh`, and the script is read-only mounted. The script is read-only verification — performs no remediation, no installs, no modifications outside writing the state file. (Acceptance criterion in Plan 02 enforces this.) Bind mount scope is limited to HOST_DATA_ROOT. |
| T-03-06 | Information disclosure | Ollama's API exposes loaded model names via `/api/tags` without auth | accept | Ollama has no native auth and the project deliberately puts auth in the router (Phase 2).
