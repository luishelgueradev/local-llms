Required behavior (D-08, D-10):

**1. Shebang and strict mode:**
- Use `#!/usr/bin/env bash` shebang.
- Use `set -uo pipefail` (NOT `set -e` — we want full diagnostic output even if early checks fail; track failures via a counter and exit at end).

**2. Constants near top of file (Claude's discretion on log style — keep readable, no color libs):**
- `readonly MODEL="llama3.2:3b-instruct-q4_K_M"`
- `readonly OLLAMA_URL="http://127.0.0.1:11434"`
- `readonly VRAM_THRESHOLD_MB=1024` (≥ 1 GB; CPU fallback shows 0; hardware-portable)
- `readonly OLLAMA_SVC="ollama"` (docker compose service name from compose.yml)
- `readonly PROMPT="What is 2+2? Answer in one short sentence."`
- `readonly KEEP_ALIVE="5m"` (pin the model in VRAM long enough for the post-generate nvidia-smi read; closes the timing race where `docker compose exec` latency could let Ollama start unloading before the VRAM check runs)

**3. Pre-flight checks inside the smoke test (defensive; not the same as preflight-gpu.sh):**
- Verify `docker compose ps --services --filter status=running` includes `ollama` — if not, fail with: "ollama service is not running. Run: docker compose up -d ollama".
- Verify `curl -fsS "${OLLAMA_URL}/api/tags"` returns a JSON object — if not, fail with: "ollama is not reachable on ${OLLAMA_URL}. Check `docker compose ps ollama` healthcheck status.".
- Verify the model is loaded: parse the JSON from `/api/tags` and check that `MODEL` appears in the `models[].name` list. If not, fail with: "model ${MODEL} is not pulled. Run: docker compose exec ollama ollama pull ${MODEL}".

**4. Step 1 — Issue a generation request to confirm Ollama actually runs the model.**
Use curl to `POST ${OLLAMA_URL}/api/generate` with body `{"model": MODEL, "prompt": PROMPT, "stream": false, "keep_alive": KEEP_ALIVE}` (build the JSON via `python3 -c 'import json; print(json.dumps(...))'` — never hand-build JSON in shell). The `keep_alive: "5m"` field tells Ollama to keep the model resident in VRAM for 5 minutes after the request returns — this prevents the timing edge where `docker compose exec` latency (Step 2 below) could otherwise miss the active VRAM window if Ollama unloaded the model on idle. Validate the response is JSON with a non-empty `response` field. On parse failure or empty content, fail with diagnostic. On success, log `[smoke-test] PASS: model returned N chars`.

**5. Step 2 — Run nvidia-smi inside the Ollama container.**
Use `docker compose exec -T "${OLLAMA_SVC}" nvidia-smi` (the `-T` flag disables TTY allocation, required for clean output capture). Capture output into `SMI_OUT`. On non-zero exit, fail with: "nvidia-smi inside the ollama container failed. This is the WSL2 silent CPU fallback signature (PITFALLS Pitfall 1). Re-run: bash bin/preflight-gpu.sh".

**6. Step 3 — Assert at least one GPU is listed.**
Grep `SMI_OUT` for the pattern `NVIDIA-SMI[[:space:]]+[0-9]+`. On miss, fail with: "no NVIDIA-SMI banner in nvidia-smi output."

**7. Step 4 — Assert an `ollama` process is bound to the GPU.**
This is the load-bearing CPU-fallback catch. Grep `SMI_OUT` (case-insensitive) for `ollama`. On miss, fail with: "no ollama process visible in container nvidia-smi. This is the silent-CPU-fallback signature: model loads but runs on CPU. Inference will be 50-100x slower than GPU. Diagnose with: bash bin/preflight-gpu.sh". With `keep_alive: "5m"` set in Step 4's request body, the model is guaranteed resident in VRAM at this point — no retry loop needed.

**8. Step 5 — Assert VRAM-in-use >= 1024 MiB.**
Parse the "FB Memory Usage / Used" field from nvidia-smi. The default text output has rows like `|   1843MiB /  16380MiB |`. Extract the first `[0-9]+MiB / [0-9]+MiB` match and take the `MiB` used number. If the parse fails (empty), fail with: "could not parse VRAM usage from nvidia-smi output." If `USED_MIB < VRAM_THRESHOLD_MB`, fail with: "VRAM in use (${USED_MIB} MiB) is below the ${VRAM_THRESHOLD_MB} MiB threshold. CPU fallback would show 0 MiB or a tiny driver-only value. The /api/generate request was sent with keep_alive=5m so the model should still be resident — if VRAM is below threshold, suspect actual CPU fallback rather than a timing race."

**9. Final summary on full pass:**
Print a banner with: model used, container nvidia-smi GPU + ollama process visible, VRAM-in-use vs threshold. End with `[smoke-test] Phase 1 GPU verification: COMPLETE.` and exit 0.

**10. CLI flags (Claude's discretion — keep minimal):**
- `-m MODEL` / `--model MODEL`: override curated model (default `llama3.2:3b-instruct-q4_K_M`).
- `--threshold MB`: override VRAM floor (default 1024).
- `-h` / `--help`: print usage and exit 0.

**Forbidden:**
- No `tokens/sec` measurement (D-10 — explicitly out of scope, brittle).
- No model pull (D-09 — manual is mandatory; the script tells the user the exact pull command if missing).
- No `:latest` references — model tag fully pinned to `llama3.2:3b-instruct-q4_K_M`.
- No background polling / retry-on-failure — fail immediately, point at preflight-gpu.sh.
