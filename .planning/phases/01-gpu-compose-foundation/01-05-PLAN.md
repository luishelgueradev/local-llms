<diagnosis-summary>
Three coherent symptoms from one structural pattern (verbatim from the debug session at `.planning/debug/preflight-gpu-wsl2-and-state-write.md`):

1. **Symptom 1 — host_nvidia_smi FAIL gates exit on WSL2.** Line 346 of `bin/preflight-gpu.sh` classifies `host_nvidia_smi` as `functional`, but the check (lines 142-151) probes only `command -v nvidia-smi` in the WSL distro's PATH. On Docker Desktop on Windows + WSL2, the Windows-side driver projects `nvidia-smi.exe` into `/usr/lib/wsl/lib/` and via `/mnt/c/Windows/System32/nvidia-smi.exe` — neither is on PATH. The Phase 1 Defect 2 rationale (01-04-SUMMARY.md) was: "container_nvidia_smi is the authoritative test if it passes, GPU passthrough works regardless of how the toolkit got configured". Yet `host_nvidia_smi` was left in the gating set when `nvidia_ctk` and `daemon_json` (with the same WSL2-host-shape failure mode) were moved to `diagnostic`.

2. **Symptom 2 — `mv: replace '/srv/local-llms/.preflight-state.json'` interactive prompt.** The `mv` alias hypothesis is **wrong** (`type mv` returns `/usr/bin/mv`, no alias). The actual cause: GNU `mv` prompts when (a) destination exists, (b) calling user cannot unlink/replace it, and (c) stdin is a TTY — regardless of `-i`. The destination is root-owned `0644` (left over from a prior in-container `gpu-preflight` run that wrote it as uid 0 through the bind mount on compose.yml line 75). **`mv -f` alone is INSUFFICIENT** because `-f` only suppresses the prompt; the underlying `unlink(2)` still fails when the calling user lacks write permission on the destination file. Lines 461, 508, 567 of `bin/preflight-gpu.sh` use the bare-`mv` pattern with no defense.

3. **Symptom 3 — abandoned `.preflight-state.json.tmp.NNNNN` files.** `grep -n "trap" bin/preflight-gpu.sh` returns zero matches. `${TMPFILE}` is created on line 461; the post-write `rm -f "${TMPFILE}"` calls on lines 512 and 571 only run when the rename returns non-zero. When `mv` blocks on a prompt and the user Ctrl-Cs, SIGINT bypasses those branches. Direct proof: two leftover `.tmp.64770` and `.tmp.65114` files observed in `/srv/local-llms/`.

**The minimum correct fix:** (a) reclassify `host_nvidia_smi` to `diagnostic` (one-line change at line 346 — mirrors `nvidia_ctk`/`daemon_json` on lines 348-349); (b) replace the python-path `mv` (line 508) with `os.replace(tmp, STATE_FILE)` inside the existing Python heredoc — `rename(2)` succeeds with directory-write permission alone, irrespective of the existing file's owner, because `/srv/local-llms/` is owned by `luis:luis` (the parent-directory unlink path); (c) replace the printf-fallback `mv` (line 567) with the same `os.replace` pattern via a tiny inline `python3 -c` (or `install -T` if python3 is unavailable — the existing fallback only triggers if python3 is missing, which is unusual); (d) register a script-scope `trap '[ -n "${TMPFILE:-}" ] && rm -f "${TMPFILE}" 2>/dev/null' EXIT INT TERM HUP` after `set -uo pipefail` (line 45), with `TMPFILE=""` declared at script scope so the unset-variable expansion under `set -u` doesn't blow up.
