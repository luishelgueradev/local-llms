<!-- Existing concrete adapters this implementation mirrors: -->

LlamacppOpenAIAdapter (router/src/backends/llamacpp-openai.ts) — almost-identical shape: composes openai SDK with baseURL, uses canonicalToOpenAIChatCompletionParams for inbound translation, openAIChatCompletionToCanonical + openAIChunksToCanonicalEvents for outbound. VLLMOpenAIAdapter copies this shape; vLLM's tool-call behavior is handled UPSTREAM via --enable-auto-tool-choice + --tool-call-parser=hermes flags (Plan 07-01 set these on the service command line). The adapter does NOT need vLLM-specific tool-call code.

OllamaOpenAIAdapter (router/src/backends/ollama-openai.ts) — Ollama's `/api/chat` native path is taken for vision (Phase 4 D-D); for vLLM there is NO native path requirement (vLLM's /v1/ is sufficient for all current capabilities chat + tools — vision lands in Phase 8 with a separate model entry on a vision-capable vLLM build, NOT this phase).

Based on 07-RESEARCH §"Pattern: vLLM Adapter — mirror LlamacppOpenAIAdapter" + Validated Code Snippets, with these methods:

import OpenAI from 'openai';
import type { BackendAdapter } from './adapter.js';
import type { CanonicalRequest, CanonicalResponse, CanonicalStreamEvent } from '../translation/canonical.js';
import { canonicalToOpenAIChatCompletionParams } from '../translation/openai-in.js';
import { openAIChatCompletionToCanonical, openAIChunksToCanonicalEvents } from '../translation/openai-out.js';

export class VLLMOpenAIAdapter implements BackendAdapter {
  private readonly client: OpenAI;
  constructor(baseURL: string) { /* OpenAI client with apiKey: 'vllm', baseURL, timeout: 60_000 */ }
  async chatCompletionsCanonical(...) { /* canonical → openai params → SDK → openai response → canonical */ }
  async chatCompletionsCanonicalStream(...) { /* canonical → openai params → SDK stream → reassemble via openAIChunksToCanonicalEvents */ }
  async probeLiveness(signal) { /* this.client.models.list, return {ok, latencyMs, error} */ }
}

NOTE: vLLM (especially the chat container) takes 600s start_period for first cold boot. The adapter's probeLiveness will return `ok: false` during that window — that is correct behavior; the liveness scheduler caches the result and /readyz returns 503 until the container is healthy. Don't add a longer probe timeout: the per-probe 2s default in liveness.ts is fine because a cold vLLM returns connection-refused immediately (not slow), and once it's warm /v1/models is fast.
