<objective>
Ship OPS-03: a disk-usage alert script that fires VISIBLY when `${HOST_DATA_ROOT}` exceeds a configurable threshold. SC3 ("a disk-usage alert fires visibly — log line, Grafana alert, or simple cron-ntfy hook") delivered via the simplest viable path: stdout log + optional HTTP hook.

## Phase Goal

**As an** operator running this stack, **I want to** be alerted (visibly) when the disk holding my models / DB / OWUI state is getting close to full, **so that** I can free space (via OPS-01 gc-models.sh, OPS-02 retention tuning, manual cleanup) BEFORE a stuck `pg-backup` write or a 14 GB GGUF download takes the stack down with `ENOSPC`.

## Why stdout log + optional ntfy instead of Grafana / Alertmanager / Prometheus

- **Grafana alerting setup is heavyweight for a single-host single-user stack.** Grafana 12.4.3 (Phase 7 OBS-04) has alerting in OSS but configuring contact points + notification policies for ONE alert is overkill. CONTEXT D-claude-discretion line 32: 'Planner picks the simpler path that meets SC3.'
- **Alertmanager is explicitly v2.** REQUIREMENTS.md ALERT-01 + CONTEXT §Deferred line 84.
- **A cron-driven script with a structured log line is observable EVERYWHERE.** journalctl, MAILTO, log aggregators (future Loki integration), `tail -F`, Grafana via Promtail — all consume stdout. The single source-of-truth output is the same.
- **The HTTP hook is opt-in and generic.** ntfy.sh is the easiest example; the same `curl -d ...` shape works with Discord / Slack / Pushover / generic webhooks. No vendor lock-in.

## Out of scope (deferred ideas / explicit non-goals)

- Inode-usage check (separate concern; rarely the failure mode on a model-storage volume — files are huge, inode count is low).
- Per-subdirectory breakdown (operator runs `du -sh ${HOST_DATA_ROOT}/*` manually; not the alert's job).
- Trend / rate-of-fill alerts (need time-series — punt to Prometheus + Grafana, v2).
- Hysteresis / cooldown to avoid alert spam (a 15-min cron cadence is the de-facto cooldown — the operator gets at most 4 alerts per hour).
- Alertmanager integration (v2).
- Auto-remediation (running OPS-01 gc-models.sh from this script) — explicitly NO: gc-models.sh requires a confirmation phrase by design; auto-running it would violate the destructive-op contract.

<!-- Inputs -->

- `HOST_DATA_ROOT` — from caller env or `.env` (grep|cut|sed extraction; same pattern as restore-drill.sh).
- `DISK_ALERT_THRESHOLD_PCT` — integer 1-99, default 80.

<!-- Output: structured single-line log on stdout -->

On NO breach (used_pct < threshold):
```
[disk-alert] LEVEL=INFO target=/srv/local-llms used_pct=42 threshold_pct=80 fs=/dev/nvme0n1p2 ts=2026-05-17T18:30:00Z hostname=local-llms-host
```

On breach (used_pct >= threshold):
```
[disk-alert] LEVEL=WARN target=/srv/local-llms used_pct=85 threshold_pct=80 fs=/dev/nvme0n1p2 ts=2026-05-17T18:30:00Z hostname=local-llms-host
```

The structured format (key=value pairs) is grep-friendly. Same shape as a future Loki log query.

<!-- Output: optional HTTP push (only on breach) -->

If `NTFY_URL` is set + non-empty AND breach fired:
```bash
curl --fail -sS --max-time 10 \
  -H 'Title: local-llms disk alert' \
  -H 'Priority: high' \
  -H 'Tags: warning,disk' \
  -d "Disk usage ${PCT}% on ${HOST} (${HOST_DATA_ROOT}); threshold ${THRESH}%" \
  "$NTFY_URL"
```

The curl is best-effort: failure does NOT fail the script. Curl exit code is logged in a SECOND log line if non-zero.

NEVER include the full NTFY_URL in any log output (the URL itself may be the credential for a private ntfy topic — T-09-I). Log only the HOST portion via sed extraction.

<!-- df parsing -->

`df -P "${HOST_DATA_ROOT}"` returns POSIX-portable output:
```
Filesystem     1024-blocks      Used Available Capacity Mounted on
/dev/nvme0n1p2   976283964 481929212 444669252      53% /
```

Parse: skip header (line 2 is data), col 5 is `Capacity` (e.g. `53%`), col 1 is `Filesystem`. Use `awk 'NR==2 {gsub("%","",$5); print $5, $1}'`.

`df -P` is intentional (not plain `df`): GNU df without `-P` may emit multi-line output for long device names, breaking the awk row-shape assumption.

<!-- Pattern reference -->

- bin/restore-drill.sh — shell-idiom reference (NOT destructive — disk-alert.sh has no confirmation gate; pure read + alert).
- bin/preflight-gpu.sh — FAILURES counter pattern + sectioned output.

<!-- Crontab integration (README documents) -->

```
# /etc/crontab or `crontab -e`
# Run every 15 minutes; log to file owned by operator.
*/15 * * * * cd /path/to/local-llms && bash bin/disk-alert.sh >> /var/log/local-llms-disk.log 2>&1
```

If the operator prefers MAILTO-style alerts (only mail on breach):
1. Set `MAILTO=admin@host` at the top of the crontab.
2. Pipe through grep so only WARN lines reach stdout: `bash bin/disk-alert.sh | grep -E '^\[disk-alert\] LEVEL=WARN'`.
3. On no-breach: grep returns 0 matches → cron emits nothing → no email. On breach: grep matches → cron mails the line.
