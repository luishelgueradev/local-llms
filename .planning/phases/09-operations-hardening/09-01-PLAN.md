<objective>
Ship OPS-01: a garbage-collection script that scans the model-file volumes against `router/models.yaml` and removes (with operator confirmation) files no longer referenced by any registry entry. Default dry-run; destructive only with `--apply` + interactive `GC` phrase or `--yes`. Move-to-trash semantics, not `rm` — preserves T-09-D safety. Allowlist-checked paths — preserves T-09-E safety.

## Phase Goal

**As an** operator running this stack long-term, **I want to** identify and remove model files on disk that no longer correspond to any entry in `router/models.yaml`, **so that** disk space doesn't bloat from old GGUFs / HF snapshots I forgot about after model swaps.

## Why this script and not `du | sort | rm`

- The set of model files I want to keep is already declared — it's in `router/models.yaml`. The GC step is the inverse: "list everything ELSE."
- Manual `rm` against a 14 GB GGUF is the wrong UX for a destructive op against a constrained resource (16 GB VRAM hardware → disk-side files matter).
- Move-to-trash is the right tradeoff: if I delete the wrong file, the next `ollama pull` re-downloads at LAN speed; if I delete the right file, I save the disk immediately. The trash dir lets me undo if the dry-run lied.

## Out of scope (deferred ideas / explicit non-goals)

- Automatic trash purge after N days (Phase 9 is manual; v2 may add).
- Cleaning up Postgres `request_log` / `usage_daily` rows that reference deleted models (DATA-02 keeps the rows; v2 ALERT-01 territory).
- Operating across multiple HOST_DATA_ROOT instances (single-host stack — single root).
- Cleaning up vLLM-compile-cache or HF-cache lockfiles (out of OPS-01 — those are inference-runtime managed).
- An HTTP admin endpoint (consistent with D-C4 from Phase 8 — psql / shell are the canonical ops paths).

Output: 1 new bash script + 1 vitest parser test + 1 README §Operations subsection. ~10-15% context.


<!-- models.yaml parsing — the source of truth for "referenced" -->

router/models.yaml has this shape (excerpt):
```yaml
models:
  - name: llama3.2:3b-instruct-q4_K_M
    backend: ollama
    backend_model: llama3.2:3b-instruct-q4_K_M
    ...
  - name: qwen2.5-7b-instruct-awq
    backend: vllm
    backend_model: Qwen/Qwen2.5-7B-Instruct-AWQ
    ...
```

Pull every `name:` value AND every `backend_model:` value. The union is the "referenced token set".

A file is REFERENCED if any path segment of its relative-to-root path contains a referenced token as a substring. Sample matches:
- `models-gguf/ollama/blobs/sha256-abc...` referenced by `llama3.2:3b-instruct-q4_K_M` ONLY IF Ollama's manifest at a sibling path lists this blob — coarse parsing cannot determine this. To stay safe, treat `models-gguf/ollama/` as a black box (always referenced, never GC-able through this script — operator uses `ollama rm <model>` from inside the container).
- `models-gguf/gguf/qwen2.5-7b-instruct-q4_K_M.gguf` referenced by `qwen2.5-7b-instruct-q4_K_M` (basename substring match).
- `models-hf/Qwen--Qwen2.5-7B-Instruct-AWQ/` referenced by `Qwen/Qwen2.5-7B-Instruct-AWQ` (HF cache replaces `/` with `--` in dir name — match the underlying ID).

For the FIRST cut, scope to two simple cases:
1. Files directly under `models-gguf/gguf/*.gguf` — match by basename substring against any token in the referenced set. Anything that doesn't match is a candidate for GC.
2. Top-level directories under `models-hf/` — the HF cache directory naming convention is `<org>--<repo>` (separator `--`). For each top-level dir, reconstruct the org/repo as `<org>/<repo>` (replace first `--` with `/`) and check if that string matches any `backend_model:` value. Unmatched dirs are candidates.

Explicitly EXCLUDE from the candidate set:
- `models-gguf/ollama/` — Ollama manages its own blob lifecycle via the container's `ollama rm`. The README documents this.
- Anything under `${HOST_DATA_ROOT}/.gc-trash/` — recursion safety.
- Hidden files / dotfiles under either root.

This is intentionally coarse — false-positives (skipping a file that COULD be GC'd) are fine; false-negatives (deleting a file that's still referenced) are not. The README documents the limitation and recommends `docker compose exec ollama ollama rm <name>` for Ollama's blob store.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → host filesystem | `bin/gc-models.sh` runs with the operator's UID/GID. If invoked with broken HOST_DATA_ROOT or via a symlink attack, it could delete unintended files. The allowlist + readlink -f checks bound the blast radius. |
| operator → trash directory | `${HOST_DATA_ROOT}/.gc-trash/` is operator-owned (bootstrap-host.sh's chown). Trash is NOT auto-purged; operator has full UNDO until they manually `rm -rf` the trash dir. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-D-01 | D (Denial of Service) | gc-models.sh fills disk during move-to-trash (atomic mv within same filesystem is O(1), but a cross-filesystem mv would copy first then delete) | mitigate | Move-to-trash uses `mv` within the SAME `${HOST_DATA_ROOT}` mount (trash dir is a sibling of models-gguf/ + models-hf/) → atomic rename, no extra disk consumed. Script asserts the trash dir resolves to the same filesystem as the source roots via `stat -c '%m'` comparison before queuing. If the operator has mounted `${HOST_DATA_ROOT}/.gc-trash/` to a different volume, the script warns + falls back to dry-run-only. |
| T-09-D-02 | D (Denial of Service) | gc-models.sh deletes a model file while inference is mid-flight against it (file open → race) | mitigate | Move-to-trash via `mv` is atomic at the directory entry level. A backend with an open file descriptor on the moved file keeps reading from the file (POSIX semantics) — the operator sees a successful GC + the active stream finishes normally.
