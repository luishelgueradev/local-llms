<objective>
Ship OPS-02: an off-host backup destination on top of the existing Phase 5 `pg-backup` sidecar's on-host dumps. Recommended path per CONTEXT D-claude-discretion: `restic` to a local-LAN or external-HDD target. Encryption + dedup built into restic; no rclone-to-cloud complexity for v1.

## Phase Goal

**As an** operator running this stack on a single host, **I want to** have my daily Postgres dumps published to an off-host destination automatically, **so that** a disk failure on the host doesn't lose both the live data AND the backups simultaneously (the standing limitation documented in README §Phase 5 §"Known limitations" — that limitation closes when this plan ships).

## Why restic and not rclone

- Restic encrypts at rest by default → BACKUP_RESTIC_PASSWORD doubles as the encryption key. Rclone leaves encryption as an opt-in `crypt` backend that adds rotation complexity.
- Restic's `forget --keep-daily/weekly/monthly --prune` is a one-line retention policy. Rclone retention requires custom scripting.
- Restic supports both LAN destinations (sftp, rest-server, local) AND object storage (b2, s3, gcs, azure) with the same CLI — operator can change destinations without changing the script.
- CONTEXT §Specifics line 74 picked restic explicitly: "Prefer `restic` over `rclone` for OPS-02 — encryption built in; simpler retention policy semantics."

## Out of scope (deferred ideas / explicit non-goals)

- Multi-destination backup (NAS + S3 in parallel) — CONTEXT §Deferred line 83 (v2 if needed).
- Encrypted `.env` at rest via sops/age/vault — CONTEXT §Deferred line 85.
- Automatic monitoring of backup freshness (Grafana panel showing "last successful off-host backup age") — OBS layer, deferrable; if the operator wants it, they add a `restic snapshots --json | jq` panel later. NOT a Plan 09-02 deliverable.
- Backing up models-gguf/ or models-hf/ — already explicitly out of scope per REQUIREMENTS.md §"Out of Scope" line 169 ("Models-volume backup — Models are re-downloadable").
- Backing up `${HOST_DATA_ROOT}/openwebui/` (OWUI's local filesystem state) — the OWUI Postgres DB is already covered by the existing dump; OWUI's filesystem state is uploaded files + temp cache. The README notes this limitation; v2 may add.
- Backing up Valkey — Valkey is cache; nothing in there outlives a restart (registry cache, rate-limit counters, idempotency state). Documented out-of-scope.
- Replacing the existing pg-backup sidecar — the sidecar produces the local dumps; this plan publishes them off-host. Two distinct concerns.

<!-- restic surface used by this script -->

```bash
# Repo URI examples (operator picks one and sets BACKUP_RESTIC_REPO):
#   sftp:backup-host:/srv/restic-repos/local-llms  ← LAN SSH target
#   /mnt/external-hdd/restic-repos/local-llms      ← local-path (external HDD)
#   b2:my-bucket-name:/local-llms                  ← Backblaze B2
#   rest:https://rest-server.lan/local-llms        ← rest-server (LAN)

# Init (operator runs ONCE — script does NOT auto-init):
restic -r "$BACKUP_RESTIC_REPO" init

# Daily run (script does this):
RESTIC_PASSWORD="$BACKUP_RESTIC_PASSWORD" restic -r "$BACKUP_RESTIC_REPO" \
  backup --tag local-llms --tag postgres --tag "$(hostname -s)" \
  --host "$(hostname -s)" \
  "${LATEST_DUMP_PATH}"

# Retention enforcement (script does this in the SAME invocation):
RESTIC_PASSWORD="$BACKUP_RESTIC_PASSWORD" restic -r "$BACKUP_RESTIC_REPO" \
  forget --keep-daily 7 --keep-weekly 4 --keep-monthly 6 --prune
```

`RESTIC_PASSWORD` is passed via env, NEVER via `--password-file` or `--password-command` (we already have the password in `.env`; the env-export pattern is simpler and keeps the password out of argv where `ps` would show it).

<!-- Crontab integration -->

README documents:
```
# /etc/crontab or `crontab -e` (host crontab, NOT inside a container)
# Run daily at HH:30, 30 minutes after the pg-backup sidecar's daily dump.
30 4 * * * cd /path/to/local-llms && bash bin/backup-postgres.sh >> /var/log/local-llms-backup.log 2>&1
```

The recipe explains: pg-backup sidecar's `while true; sleep 86400` loop starts when the sidecar first comes up — if compose was started at 04:00 UTC, dumps land at ~04:00 UTC every day. Schedule the off-host publish ~30 min later to ensure the day's dump is on disk. Operator tunes for their stack-start time.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host shell → restic repo | restic publishes encrypted blobs to a destination the operator controls. The repo URI + password are the trust anchors — anyone with both can read every snapshot. |
| `.env` file → script subprocess | BACKUP_RESTIC_PASSWORD flows via env var to the restic subprocess. Script does NOT log it. Anyone with `.env` read access can decrypt the backups. |

## STRIDE Threat Register
