## Phase Goal

**As an** operator running this stack who needs to rotate `ROUTER_BEARER_TOKEN` (suspected exposure, periodic rotation, post-incident hygiene), **I want to** a step-by-step procedure that updates every consumer of the token — including Open WebUI's persisted connection — WITHOUT breaking the OWUI admin user or chat history, **so that** I can rotate confidently in 5-10 minutes without losing data.

## Why this is docs-only and not a script

- The rotation needs operator decisions at multiple steps (regenerate vs reuse, which OWUI path, when to discard the old token). A script would either: (a) require a chain of confirmation gates that defeat its automation purpose, or (b) make destructive choices on the operator's behalf. The README-as-script-substitute pattern is the right tradeoff for single-user infra.
- The OWUI path has two valid variants (admin UI vs direct SQL). A single rotation script would force the operator to pick one in advance. Documenting both lets the operator pick at rotation time.
- The verification step is grep-based — trivially scriptable, but more useful as a snippet the operator can copy-tweak (e.g. they might want to grep last 72h, not 24h).

## Out of scope (deferred ideas / explicit non-goals)

- Automated bearer rotation tooling (CONTEXT §Deferred line 87 — v2).
- Encrypted .env at rest (CONTEXT §Deferred line 85 — v2; sops / age / vault are the v2 candidates).

- Rotation of TRAEFIK_BASIC_AUTH / OWUI_SECRET_KEY — those have their own irreversibility constraints (CONTEXT §Phase 6 — OWUI_SECRET_KEY rotation breaks at-rest-encrypted OWUI fields). Out of OPS-04.

<!-- The token consumers (pre-flight inventory the README documents) -->

`ROUTER_BEARER_TOKEN` is consumed by:

1. **The router itself** (`router/src/auth/bearer.ts`) — reads from `.env` via env-loader; constant-time compares against incoming `Authorization: Bearer ...` header.
2. **Open WebUI** (compose.yml:968 — `OPENAI_API_KEYS=${ROUTER_BEARER_TOKEN}`) — seeded at OWUI first boot, then PERSISTED to the `openwebui` Postgres DB via OWUI's PersistentConfig mechanism. Subsequent restarts read from DB, NOT from env. This is the path that requires the special handling — restarting the OWUI container with a new `OPENAI_API_KEYS` env value does NOT update the stored connection.
3. **Smoke tests** (`bin/smoke-test-router.sh`, `bin/smoke-test-cloud.sh`, `bin/smoke-test-traefik.sh`) — all read `ROUTER_BEARER_TOKEN` from `.env` at invocation time. No persistence outside `.env`.
4. **The operator's agent clients** (Claude Code, cline, llm CLIs, custom scripts, etc.) — wherever the operator put it. README cannot enumerate; the procedure asks the operator to do this inventory.
5. **The operator's shell history** (if they ever `curl -H "Authorization: Bearer $TOKEN" ...`) — bash history file. README mentions a `history -d` cleanup step.

<!-- Verification grep recipe (post-rotation) -->

```bash
# After rotation, capture the OLD token's prefix (first 8 hex chars) BEFORE
# discarding it. Use this prefix for the grep — never the full token.
OLD_PREFIX="abc12345"  # operator pastes — first 8 chars of old token

# Then grep across the three log surfaces:
for svc in router openwebui traefik; do
  echo "=== $svc ==="
  docker compose logs "$svc" --since 24h 2>&1 | grep -c "$OLD_PREFIX" || echo 0
done
# Expected: 0 0 0 (all three services emit zero matches).
# Non-zero: investigate; likely a non-redacted log statement OR an OWUI-side
# log that captured the env value before first boot persisted it (acceptable
# for OWUI first-boot lines older than the rotation timestamp).
```

Pino redaction (Phase 2 ROUTE-05 + Plan 02-CONTEXT D-redaction) means the router should emit ZERO Authorization-header matches across the entire log history. The grep is a regression check, not a discovery probe.
