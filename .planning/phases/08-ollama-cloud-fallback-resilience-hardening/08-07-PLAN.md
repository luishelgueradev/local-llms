<objective>
Implement the Idempotency-Key multiplexer per ROUTE-12 + D-D5 + D-D6. Concurrent retries with the same key receive the same response — only ONE upstream generation runs, regardless of follower count. Closes PITFALLS Pitfall 14 (retry storms DoS the local GPU) for SDK-driven retries that send the header.

## MVP slice user story

An agent SDK with default retry-on-failure fires 8 concurrent retries of the same prompt (same Idempotency-Key) after a transient network blip. Without this plan, the local GPU runs 8 generations and the operator's request_log grows by 8 rows for a single user intent. With this plan, ONE generation runs; 8 responses are emitted from the same upstream answer; request_log has 8 rows but all share `upstream_message_id`, so cost-attribution dashboards (Plan 08-08) collapse them to one logical request.

## Why Valkey pub/sub + cached chunks

Two replay strategies for followers:
- **Pure pub/sub**: follower subscribes to the channel. Misses chunks emitted before subscribe.
- **Pure cache + poll**: follower reads chunks list, polls for 'done' marker. Slow / high traffic on Valkey.
- **HYBRID (chosen)**: follower SUBSCRIBEs FIRST (now bound to receive all future chunks), THEN reads the cached chunks list (covers chunks already emitted), THEN if 'done' is in the result key, replay cached + result and exit; else continue piping channel events to the response. This avoids both the missed-chunk race AND the poll spam.

## Order of operations (leader)

1. SETNX lock key with TTL 30min → role: leader.
2. Call adapter (stream or non-stream).
3. For each canonical event (stream) or final result (non-stream): SERIALIZE to JSON, RPUSH to chunks list, PUBLISH to channel.
4. On adapter completion: SETEX result key with serialized terminal payload, EXPIRE chunks list to 15min, PUBLISH 'done' marker.
5. On adapter ERROR (5xx, timeout): SETEX result key with serialized error envelope, PUBLISH 'error' marker. Followers receive the same error.
6. On leader CLIENT DISCONNECT (req.raw.on 'close'): DEL the lock key (or wait for TTL — DEL is cleaner); upstream is aborted (existing AbortController plumbing). Followers receive 'disconnect' marker; they disconnect too.

## Order of operations (follower)

1. lock SETNX FAILED → role: follower (lock already held by leader).
2. SUBSCRIBE to channel BEFORE reading anything cached.
3. LRANGE chunks list 0 -1 → replay cached chunks via reply.sse(...) (stream) or buffer (non-stream).
4. GET result key:
   - If 'done': finalize the response (stream: send [DONE] + close; non-stream: send the buffered + parse body).
   - If not yet present: continue piping subscribe events to the response until 'done' marker arrives.
5. On 30s SUBSCRIBE timeout without 'done': respond with 504 + envelope `idempotency_timeout` (rare — leader hung).

## Out of scope

- Cross-router multiplexing (multi-instance Valkey shared state — works automatically, but not v1 deployed).
- Idempotency on `/v1/messages/count_tokens` (no upstream call; CPU-only; key irrelevant).
- Idempotency-Key on `/v1/models` (read-only; idempotent by nature).
- Persistent idempotency cache beyond Valkey TTL (15 min is the v1 ceiling — D-D6 explicitly).

```typescript
// router/src/middleware/idempotencyKey.ts — Plan 08-07.
// Helper for validating and extracting the Idempotency-Key header.
import { InvalidIdempotencyKeyError } from '../errors/envelope.js';

const KEY_RE = /^[A-Za-z0-9._:-]{1,256}$/;

export function extractIdempotencyKey(headers: Record<string, string | string[] | undefined>): string | undefined {
  const raw = headers['idempotency-key'];
  if (raw === undefined) return undefined;
  if (Array.isArray(raw)) {
    // Duplicate header — treat as invalid.
    throw new InvalidIdempotencyKeyError(raw.join(','));
  }
  if (!KEY_RE.test(raw)) {
    throw new InvalidIdempotencyKeyError(raw);
  }
  return raw;
}
```

<!-- resilience/idempotency.ts — multiplexer module -->

The module exports `makeIdempotencyMultiplexer({ valkey, log, subscriberFactory }): IdempotencyMultiplexer` with the following surface:

```typescript
export interface IdempotencyMultiplexer {
  /**
   * Acquire lock for this key. Returns 'leader' if SETNX succeeded; 'follower'
   * if the key already exists. The lock has a 30-min ceiling TTL (Valkey side);
   * the data TTL is bumped to 15 min on stream end via finalize().
   */
  acquire(key: string, requestId: string): Promise<{ role: 'leader' | 'follower' }>;

  /**
   * Leader API — non-stream path. Cache the JSON body + publish 'done'.
   * The body is serialized as JSON; followers parse and re-emit. Final TTL is
   * 15 min (D-D6) via EXPIRE.
   */
  publishNonStream(key: string, body: unknown, upstreamMessageId?: string): Promise<void>;

  /**
   * Leader API — stream path, per-event. Serialize event as JSON, RPUSH to
   * the chunks list, PUBLISH to the channel. Called once per canonical event.
   */
  publishStreamEvent(key: string, event: unknown): Promise<void>;

  /**
   * Leader API — stream path, finalize. PUBLISH 'done' (or 'error' / 'aborted'),
   * set EXPIRE on chunks + result keys to 15 min.
   */
  finalizeStream(key: string, status: 'done' | 'error' | 'aborted', upstreamMessageId?: string, finalPayload?: unknown): Promise<void>;

  /**
   * Follower API — non-stream path. SUBSCRIBE to channel, GET result; if
   * 'done' marker present, return the cached body; else block until 'done'
   * or 30s timeout. Returns the parsed body for the route to reply.send().
   */
  awaitNonStreamResult(key: string, requestId: string): Promise<{ body: unknown; upstreamMessageId?: string }>;

  /**
   * Follower API — stream path. SUBSCRIBE to channel, LRANGE cached chunks,
   * yield AsyncIterable<unknown> (canonical events) for the route to translate
   * back to SSE. The iterable completes when 'done' / 'error' / 'aborted' marker
   * is observed (in cache or via subscribe).
   */
  awaitStreamResult(key: string, requestId: string): AsyncIterable<{ event?: unknown; terminal?: 'done' | 'error' | 'aborted' }>;
}
```

Implementation sketch (skeleton — refine in Task 1):

- Lock key: `idempotency:${key}:lock` (SETNX EX 1800)
- Result key: `idempotency:${key}:result` (JSON body or { status: 'done'|'error'|'aborted', upstreamMessageId, payload })
- Chunks list: `idempotency:${key}:chunks` (RPUSH each event JSON; LRANGE for follower replay)
- Channel: `idempotency:${key}:channel` (PUBLISH event JSON or 'done'/'error'/'aborted' string)
- Terminal-marker discrimination: a special JSON shape `{"$terminal":"done","upstreamMessageId":...}` distinguishes from canonical events (which never use `$terminal` field).

The subscriber for the channel needs a SEPARATE ioredis connection (ioredis docs: a connection in subscribe mode cannot issue other commands). The `subscriberFactory` opt is a `() => ValkeyClient` that returns a fresh connection for each follower (closed after use). Production wiring (in buildApp) duplicates `opts.valkey` via `valkey.duplicate()`.
