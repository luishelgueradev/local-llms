<objective>
Add the `cloud_spend_daily` Postgres view per CLOUD-05 + D-C3. Per-day breakdown of cloud generation_duration_ms (SUM(latency_ms) for backend='ollama-cloud') with distinct-generation counting via upstream_message_id.

## MVP slice user story

After Plan 08-07 ensures follower idempotent requests share the leader's upstream_message_id, this view becomes meaningful: the operator can run a single SQL query to see (a) total daily request count, (b) total daily latency ms (cost proxy), (c) distinct cloud generations (collapsing follower retries). Three numbers, one query, no admin UI, no metrics-backend.

The view is a thin DB-side projection — no router code change required. Phase 5's buffered writer already populates request_log with backend + latency_ms + upstream_message_id, so the view literally just exists at the DB layer.

## Why a VIEW instead of a materialized table

The existing usage_daily table (Phase 5 D-G2) is materialized + scheduled because it aggregates ALL request_log rows (heavy). cloud_spend_daily filters to `backend = 'ollama-cloud'` which (in this single-operator stack) is going to be tens to hundreds of rows per day — querying the view live is cheaper than maintaining a materialized table. If cloud traffic ever grows to millions/day, a materialized variant is a one-line follow-up.

## Out of scope

- USD cost calculation (COST-01 — explicit v2 backlog per CONTEXT line 116).
- Per-model breakdown (could be added as `, model` to GROUP BY but D-C3 keeps it at day-level only).
- HTTP admin endpoint (D-C4 rejected).
- Spend alerting / thresholds (out of v1; Phase 9 / v2 ALERT-01 territory).

router/src/db/migrations/008_cloud_spend_daily.sql (new file):

```sql
-- Migration 008: cloud_spend_daily view (Plan 08-08 / CLOUD-05 / D-C3).
--
-- Per-day breakdown of cloud-backend request activity. Surface for ad-hoc
-- ops queries: `docker compose exec postgres psql -U app -d router \
--   -c "SELECT * FROM cloud_spend_daily LIMIT 30;"` (D-C4).
--
-- spend_ms = SUM(latency_ms) over backend='ollama-cloud' rows. This is the
-- generation_duration_ms proxy: latency_ms includes the full request
-- lifecycle including SDK round-trip + upstream generation, which is what
-- Ollama Cloud bills GPU-time against.
--
-- distinct_generations counts unique upstream_message_id values, so the
-- N+1 follower retries from Plan 08-07's idempotency multiplexer collapse
-- to 1. The N+1 retries STILL contribute to spend_ms via their latency_ms
-- (the followers' latency is upper-bounded by the leader's stream duration,
-- but the rows exist regardless) — so distinct_generations is the better
-- "unique billable cost units" signal.
--
-- Idempotent: CREATE OR REPLACE VIEW. Re-running this migration is a no-op.

CREATE OR REPLACE VIEW cloud_spend_daily AS
SELECT
  date_trunc('day', ts)              AS day,
  COUNT(*)                           AS request_count,
  SUM(latency_ms)                    AS spend_ms,
  COUNT(DISTINCT upstream_message_id) AS distinct_generations,
  AVG(latency_ms)::int               AS avg_latency_ms
FROM request_log
WHERE backend = 'ollama-cloud'
GROUP BY 1
ORDER BY 1 DESC;

COMMENT ON VIEW cloud_spend_daily IS
  'CLOUD-05 (Plan 08-08): per-day cloud-backend request stats. spend_ms = SUM(latency_ms) = generation_duration_ms proxy. distinct_generations = COUNT(DISTINCT upstream_message_id) = unique billable units (collapses Plan 08-07 follower retries).';
```
