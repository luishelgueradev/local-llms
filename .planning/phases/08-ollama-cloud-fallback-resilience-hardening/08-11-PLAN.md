waitUntilReady shape to implement (mirrors idempotency.ts L241-267 exactly):
```typescript
export async function waitUntilReady(
  client: ValkeyClient,
  timeoutMs = 2000,
): Promise<void> {
  // Short-circuit: mock clients and already-connected real clients skip the wait.
  const c = client as unknown as {
    status?: string;
    once?: (event: string, cb: (...a: unknown[]) => void) => unknown;
    removeListener?: (event: string, cb: (...a: unknown[]) => void) => unknown;
  };
  if (typeof c.once !== 'function' || c.status === 'ready') return;
  await new Promise<void>((resolve, reject) => {
    const timer = setTimeout(() => {
      c.removeListener?.('ready', onReady);
      c.removeListener?.('error', onError);
      resolve(); // FAIL-OPEN: timeout → resolve (not reject) so boot continues
    }, timeoutMs);
    const onReady = (): void => {
      clearTimeout(timer);
      c.removeListener?.('error', onError);
      resolve();
    };
    const onError = (err: unknown): void => {
      clearTimeout(timer);
      c.removeListener?.('ready', onReady);
      reject(err instanceof Error ? err : new Error(String(err)));
    };
    c.once?.('ready', onReady);
    c.once?.('error', onError);
  });
}
```

CRITICAL distinction from idempotency.ts version: the timeout branch calls RESOLVE (not reject).
The idempotency subscriber timeout is a hard error (a wedged subscriber must not proceed).
The boot-path timeout is FAIL-OPEN — a Valkey that never becomes ready must not block startup;
the existing try/catch in the registryCache calls + file-load fallback handles the failure.
The idempotency path must keep its REJECT-on-timeout behavior. When refactoring idempotency.ts,
pass `rejectOnTimeout: true` as a parameter OR keep the idempotency await-ready block as a
`waitUntilReady` call with the timeout resolving into a thrown error. See Task 1 note below.
