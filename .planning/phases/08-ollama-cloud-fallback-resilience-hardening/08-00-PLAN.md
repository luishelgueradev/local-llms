<objective>
Close the Phase 8 precondition flagged in 07-REVIEW-FIX.md §CR-02 (SKIPPED, deferred to this phase).

Two coupled fixes:

1. **Registry-level invariant** — Add a `superRefine` clause to `RegistrySchema` that rejects any models.yaml where the same `backend_url` is declared under two distinct `backend` values. This is the structural guarantee Phase 8 needs before `OllamaCloudAdapter` ships: two backends (e.g. `ollama` and `ollama-cloud`) MAY coexist, but they cannot collide on the same URL (which would make the `(url → backend)` lookup ambiguous in the liveness probe path). Today's models.yaml does not violate this (every backend has a unique url), so this is a tightening of the contract with no operator-visible behavior change unless someone makes a typo.

2. **Probe-adapter cache key widening** — Change `router/src/app.ts probeAdapterFor` to (a) key its cache by `${backend}|${url}` instead of bare `url`, and (b) find the `ModelEntry` by both `backend` and `backend_url` instead of by `backend_url` alone. The function now requires a `backend` arg in addition to `url`; the scheduler's probe callback resolves the backend from the registry before calling probeAdapterFor.

Purpose: Land CLOUD-01's structural prerequisite ahead of Wave 1 so subsequent plans can introduce `'ollama-cloud'` as a registry-recognized backend WITHOUT inheriting the URL-ambiguous behavior the 07-REVIEW review explicitly called out as a Phase 8 blocker.

Output: Three router source edits + two regression tests. No compose.yml, no models.yaml, no env, no .env.example. Single-task plan; ~10% context — small but a hard precondition for every plan in Waves 1+.
</objective>

<!-- registry.ts widening — RegistrySchema.superRefine block -->

The current superRefine in registry.ts (lines 56–73) only validates the VRAM envelope. ADD a SECOND superRefine clause (or extend the existing one — single-superRefine, two checks is acceptable) that asserts no two distinct (backend, backend_url) pairs collide on the same URL across different backend values.

Pseudocode of the check:

  // Build url → Set<backend> map; flag any url whose Set has size > 1.
  const urlToBackends = new Map<string, Set<string>>();
  for (const m of reg.models) {
    let bs = urlToBackends.get(m.backend_url);
    if (!bs) { bs = new Set(); urlToBackends.set(m.backend_url, bs); }
    bs.add(m.backend);
  }
  for (const [url, backends] of urlToBackends) {
    if (backends.size > 1) {
      ctx.addIssue({
        code: 'custom',
        path: ['models'],
        message: `Config error: backend_url "${url}" is shared by backends [${[...backends].sort().join(', ')}]. Each backend value must have a unique URL — two backends serving the same URL makes URL→backend lookup ambiguous in the liveness probe path. Resolution: give one backend a distinct upstream URL (e.g. proxy alias) or merge the two entries under a single backend value.`,
      });
    }
  }

Permitted by the contract: multiple ModelEntries with the SAME backend AND the SAME backend_url (today's models.yaml has this — three `backend: ollama` models all point at http://ollama:11434/v1). The check only fires when the SAME url appears under DISTINCT backend values.

<!-- app.ts probeAdapterFor widening -->

Current shape (app.ts:244-255):

  const probeAdapters = new Map<string, ReturnType<typeof defaultMakeAdapter>>();
  const probeAdapterFor = (url: string) => {
    let a = probeAdapters.get(url);
    if (!a) {
      const reg = opts.registry.get();
      const entry = reg.models.find((m) => m.backend_url === url);
      if (!entry) throw new Error(`No registry entry for URL "${url}"`);
      a = defaultMakeAdapter(entry);
      probeAdapters.set(url, a);
    }
    return a;
  };

NEW shape:

  const probeAdapters = new Map<string, ReturnType<typeof defaultMakeAdapter>>();
  // Key shape: `${backend}|${url}`. The combined key disambiguates Phase 8's
  // `ollama-cloud` (https://ollama.com/v1) from any other future backend that
  // could conceivably share a URL. The registry's superRefine guarantees no
  // two distinct backends share a URL, but this cache key shape is the
  // belt-and-suspenders runtime guarantee.
  const probeAdapterFor = (backend: string, url: string) => {
    const key = `${backend}|${url}`;
    let a = probeAdapters.get(key);
    if (!a) {
      const reg = opts.registry.get();
      const entry = reg.models.find((m) => m.backend === backend && m.backend_url === url);
      if (!entry) throw new Error(`No registry entry for (backend "${backend}", URL "${url}")`);
      a = defaultMakeAdapter(entry);
      probeAdapters.set(key, a);
    }
    return a;
  };

The scheduler call site (app.ts:294-301) currently calls `probeAdapterFor(url)` from inside the `probe: async (url, signal) => {...}` callback. The new call site MUST first resolve the backend from the registry by url. Pattern:

  probe: async (url, signal) => {
    if (pgProbe && url === POSTGRES_PROBE_URL) return pgProbe(url, signal);
    const reg = opts.registry.get();
    const entry = reg.models.find((m) => m.backend_url === url);
    if (!entry) {
      // Unknown URL on the probe schedule — log + return a synthetic "down" probe
      // result rather than throw (the scheduler doesn't unwrap thrown errors here).
      return { ok: false, latencyMs: 0, error: `no registry entry for url "${url}"` };
    }
    const adapter = probeAdapterFor(entry.backend, url);
    return adapter.probeLiveness(signal);
  },

Why the url-first lookup is still safe: the liveness scheduler's URL set is derived from `registry.get().models.map((m) => m.backend_url)`; for any URL in that set, exactly one (backend, url) tuple exists per the new superRefine invariant — so the `.find()` is deterministic.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → models.yaml | A typo (two backends declared with the same url) would silently route the wrong adapter at runtime; the superRefine moves the failure to boot-time so it is loud, not silent. |
| internal → probeAdapters cache | Cache poisoning is not in the threat surface (single-process, no remote write path); the cache-key widening is for correctness, not security. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-T-00 | T (Tampering) | models.yaml — operator typo or attacker with disk write reorders backend_url to collide cloud + local | mitigate | superRefine refuses to boot when two distinct backends share a URL. The error message is explicit + actionable, so the operator can correct rather than try to "make it work". |
| T-08-S-00 | S (Spoofing) | probeAdapterFor — without the widening, two backends sharing a URL could be probed by whichever adapter happened to be cached first, leading to false-positive liveness for the non-cached backend | mitigate | Composite cache key + (backend, url) entry lookup; an `ollama-cloud` adapter can never be returned when the caller asked for `ollama` and vice versa. |
| T-08-R-00 | R (Repudiation) | unknown-url scheduler probe — silently throwing would corrupt /readyz aggregation | mitigate | The scheduler probe returns a synthetic `{ ok: false, error: 'no registry entry for url ...' }` instead of throwing; /readyz then reports the unknown url as down (correct, auditable behavior). |
