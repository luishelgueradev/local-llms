<objective>
Add the `X-Model-Backend` response header per ROUTE-10. Every successful chat / messages / embeddings response advertises which backend served it (`ollama`, `llamacpp`, `vllm`, `vllm-embed`, `ollama-cloud`).

## MVP slice user story

After Plan 08-02, an operator can curl a cloud model and get a real response — but the agent can't tell whether it was served locally or remotely without inferring from the model name. This plan closes that observability gap: `curl -I` (or curl -N piped to head) shows `X-Model-Backend: ollama-cloud` in the response headers. An agent that wants to log "served by ollama-cloud" for cost-attribution, or to fall back to a different model on cloud-only failure, now has a wire-level signal to read.

This is a thin slice — one onSend hook + three one-line stamps in route files + one module augmentation + one integration test. Wave 2 placement (after 08-02) ensures cloud entries already exist in the registry to be advertised.

## Out of scope

- Stream-branch header verification — Plan 08-10's smoke does this against a live router with curl -N.
- Header on /healthz / /readyz / /metrics / /v1/models / /v1/messages/count_tokens — those routes don't have a resolved backend (or have multiple, for /v1/models). The onSend hook's guard against `undefined req.resolvedBackend` keeps these clean.
- Mid-stream backend swap (a future feature where the router falls back to a different backend on local failure) — not in v1. Single backend per request; X-Model-Backend reflects the resolve-time choice.

Output: 1 onSend hook + 1 module augmentation + 3 one-line stamps + 1 integration test. Smallest plan in Wave 2; ~10% context.
</objective>

<!-- onSend hook in app.ts -->

In buildApp(), AFTER the existing onClose hook and BEFORE the routes are registered (around line 376, after the `// Routes` section header), add:

  /**
   * Plan 08-03 (ROUTE-10) — X-Model-Backend response header.
   *
   * onSend fires after the route handler returns and BEFORE the body is
   * serialized + flushed. For non-stream responses, this places the header
   * cleanly in the initial HTTP response. For SSE streams (chat-completions,
   * messages stream branch), the SSE plugin (fastify-sse-v2) flushes headers
   * on the first reply.sse(...) call AFTER all onSend hooks have run — so
   * the header is in the initial response block, BEFORE the first `data:` frame.
   *
   * Skip the header when req.resolvedBackend is undefined: pre-resolve errors
   * (unknown model, missing bearer) and routes that never resolve a backend
   * (/healthz, /readyz, /metrics, /v1/models, /v1/messages/count_tokens) all
   * naturally produce responses without the header.
   *
   * D-E2: Traefik passes custom response headers through by default — Plan 08-10's
   * smoke verifies this end-to-end via curl against the Traefik edge.
   */
  app.addHook('onSend', async (req, reply, payload) => {
    const backend = (req as { resolvedBackend?: string }).resolvedBackend;
    if (backend) {
      void reply.header('X-Model-Backend', backend);
    }
    return payload;
  });

The hook is async (Fastify v5 contract — `void reply.header(...)` because Fastify's reply.header returns the FastifyReply for chaining; we don't chain). The `payload` arg is the unmodified body — the hook returns it as-is.

Place the hook BEFORE the route registration block (lines 376+) so it's wired before any route can fire it. Order: agentIdPreHandler (preHandler) → route handler → onSend (this hook) → body serialize.
