## Goal Achievement

The phase goal is **partially achieved**. The data path is in place — bufferedWriter, /metrics, X-Agent-Id middleware, postgres compose service, pg-backup sidecar, restore drill, usage_daily scheduler, /readyz postgres probe all exist and are wired through buildApp. The 473/475 vitest suite passes. However, the observability seam has three load-bearing defects identified by 05-REVIEW.md that the user explicitly elected to verify before promoting the phase:

1. **CR-01 (BLOCKER):** Hot-reload of models.yaml silently kills the postgres /readyz probe — the operator must restart the router to recover from any models.yaml edit while observing /readyz.
2. **CR-02 (BLOCKER):** Streaming pre-stream errors emit zero request_log rows + zero Prometheus metric observations — the "every request gets a row" promise of SC2 / D-D4 is broken for an entire failure class.
3. **CR-03 (BLOCKER):** Mid-stream upstream errors are recorded as `status_class='success'` with NULL error_code / error_message — the observability seam reports success for what the wire correctly delivered as a failure.

These are not stub-or-missing issues — every artifact exists, is wired, and works for the happy path. The defects are in the error-path control flow. They are precisely the conditions under which "debug a runaway agent" becomes load-bearing, so they invalidate the phase goal's stated motivation ("the per-agent identity that the user will need to debug runaway agents") for the failure modes the operator most needs to diagnose.
