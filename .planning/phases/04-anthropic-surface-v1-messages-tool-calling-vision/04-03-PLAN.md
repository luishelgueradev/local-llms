CanonicalStreamEvent → SSE mapping (RESEARCH.md Pattern 2, lines 316–346):
- `{type:'message_start', message}` → `{event:'message_start', data: JSON.stringify({type:'message_start', message})}`
- `{type:'content_block_start', index, content_block}` → `{event:'content_block_start', data: JSON.stringify({type:'content_block_start', index, content_block})}`
- `{type:'content_block_delta', index, delta}` → `{event:'content_block_delta', data: JSON.stringify({type:'content_block_delta', index, delta})}`
- `{type:'content_block_stop', index}` → `{event:'content_block_stop', data: JSON.stringify({type:'content_block_stop', index})}`
- `{type:'message_delta', delta, usage}` → `{event:'message_delta', data: JSON.stringify({type:'message_delta', delta, usage})}`
- `{type:'message_stop'}` → `{event:'message_stop', data: JSON.stringify({type:'message_stop'})}`
- `{type:'ping'}` → `{event:'ping', data: JSON.stringify({type:'ping'})}`

Anthropic mid-stream error frame (D-F5, RESEARCH.md Example C):

BackendAdapter signature extension (Issue #6 resolution — supersedes the Plan 03 route-level rewriteInputTokens wrapper):
```typescript
import type { CanonicalRequest, CanonicalResponse, CanonicalStreamEvent } from '../translation/canonical.js';

export interface BackendAdapter {
  chatCompletionsCanonical(canonical: CanonicalRequest, signal: AbortSignal): Promise<CanonicalResponse>;
  chatCompletionsCanonicalStream(
    canonical: CanonicalRequest,
    signal: AbortSignal,
    opts?: { inputTokensHint?: number }
  ): Promise<AsyncIterable<CanonicalStreamEvent>>;
  probeLiveness(signal: AbortSignal): Promise<{ ok: boolean; latencyMs: number; error?: string }>;
}
```
- `inputTokensHint` is the upstream's prompt token estimate computed via `countTokens(canonical)` (Plan 02's helper) BEFORE the adapter call. The route in `messages.ts` computes it once and passes it through; the route NEVER intercepts/mutates events.
- Both adapter implementations consume `opts?.inputTokensHint` and forward it into `openAIChunksToCanonicalEvents({..., inputTokensHint: opts?.inputTokensHint ?? 0})` and/or `ollamaNativeChunksToCanonicalEvents({..., inputTokensHint: opts?.inputTokensHint ?? 0})`.
- This isolates the "input_tokens placement" concern entirely inside the adapter→translator pipeline; the route's only role is to compute the hint.

Open Question 1 resolution: the synthetic `message_start.usage.input_tokens` is set to `opts.inputTokensHint ?? 0` at emission time inside `openAIChunksToCanonicalEvents`. No back-patching. Anthropic's documented ±5% expectation per FINDING 2.2 is satisfied because countTokens uses gpt-tokenizer/cl100k_base (D-E1) — close enough to upstream's actual prompt_tokens.

Heartbeat variant (D-B5 + PATTERNS.md "Anthropic ping heartbeat payload variant" — No Analog Found row):
