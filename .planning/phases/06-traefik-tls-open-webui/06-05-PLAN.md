<objective>
Close the D-C6 BLOCKER documented in `06-VERIFICATION.md` `gaps_found[0]`: Open WebUI can currently reach `http://ollama:11434/api/tags` directly because both services share the `app` Docker network. This bypasses the D-C6 anti-bypass contract from 06-CONTEXT.md and violates ROADMAP SC5 (literal text: "no OWUI bypass connections to backends **exist**" — not just "configured").

The fix is OPTION A from the verification report (the user-chosen mitigation): introduce a new isolated `webui-app: { internal: true }` Docker network whose only members are `router`, `openwebui`, and `traefik`. Remove `app` from openwebui's network list. Ollama stays on `app + backend` per Phase 1 D-13 (the egress-fix-for-`ollama pull` constraint that drives ollama's dual-network membership — **this constraint is NOT touched by this plan**). Result: openwebui and ollama no longer share any network; the TCP route from openwebui to ollama:11434 ceases to exist; the D-C6-bypass curl returns a connection-class error instead of a JSON payload.

Plan 06-05 is a Wave 5 single-file mutation plus a regression-gate extension to `bin/smoke-test-traefik.sh`. Plans 06-01 through 06-04 stay frozen (predecessors landed; their SUMMARYs are immutable). No `.env` changes. No Traefik static/dynamic config changes. No OWUI image / env / persistence changes. No router code changes. No bearer.ts cleanup (already deferred to operator sign-off per 06-04 Hand-off step 8).

Purpose:
- **D-C6 closure at the network plane.** Move the contract from "configured" (Plan 06-03's `ENABLE_OLLAMA_API=False` env defense-in-depth) to "structurally impossible" (no shared TCP plane). Operator-as-attacker becomes irrelevant because the route simply does not exist.
- **WEBUI-04 spirit (anti-bypass).** ROADMAP SC5 says bypass connections must not exist; this plan makes that literally true.
- **Preserve Phase 1 D-13 egress design.** Ollama keeps its `app` membership (which is its egress path to `registry.ollama.ai` for `ollama pull` — the `backend` network is `internal: true` and blocks that). This plan does NOT modify ollama's network membership in any way.
- **Preserve all Phase 6 wiring.** Traefik still discovers router-edge via the `app` network (router stays on `app`); Traefik now also discovers webui-edge via the new `webui-app` network (openwebui's label is updated to point there; Traefik joins the new network). All six Traefik labels per service are preserved otherwise.
- **Regression gate.** Add the D-C6-bypass curl as a FAIL_IF_REACHABLE assertion in `bin/smoke-test-traefik.sh` so any future regression (e.g., a hypothetical Phase 7 change that re-adds `app` to openwebui) is caught by the smoke immediately.

Output:
- `compose.yml`: top-level `networks:` gains `webui-app: { internal: true }`; `router.networks` gains `webui-app`; `openwebui.networks` swaps `app` → `webui-app`; `openwebui.labels` `traefik.docker.network` updated to `${COMPOSE_PROJECT_NAME:-local-llms}_webui-app`; `traefik.networks` gains `webui-app`.
- `bin/smoke-test-traefik.sh`: new Section 5 with D-C6 regression gate; runs in `--quick` (it's fast).
- Acceptance: `docker compose up -d --force-recreate router openwebui traefik` brings the stack back healthy; `docker compose exec openwebui curl --max-time 3 http://ollama:11434/api/tags` returns connection-class error (exit code 6/7/28); `docker compose exec openwebui curl --max-time 5 -H "Authorization: Bearer $ROUTER_BEARER_TOKEN" http://router:3000/v1/models` returns JSON; `bash bin/smoke-test-traefik.sh --quick` exits 0.
